\begin{abstract}
Automatic generation of executable code from scientific papers is crucial
for both research aim and the practical adoption of
new methods in industry. Existing paper-to-code agents can produce
repositories that appear faithful and may score highly under
LLM-as-a-judge evaluation, but such apparent quality does not
ensure runnability or scientific reproducibility. We propose a
rubric- and execution-guided paper-to-code framework that
verifies generated repositories through intermediate-value
tests, runtime checks, and benchmark-level reproduction
criteria. Our workflow parses papers into structured JSON,
generates initial repositories through a planning-analysis-coding
pipeline, constructs paper-specific rubrics and \texttt{pytest}
tests, extracts runtime requirements, and packages the resulting
task into a Harbor-style evaluation environment. On 22
PaperBench papers, our prototype substantially improves
runnability and reveals a large gap between executable
repositories and scientifically reproducible repositories. These
results suggest that paper-to-code evaluation should move beyond
LLM-as-a-judge scores toward execution-grounded verification of
algorithmic and experimental fidelity.
\end{abstract}

\section{Introduction}

Reproducibility is essential for scientific progress, allowing
researchers to verify findings, build on previous work, and push the
boundaries of knowledge \citep{collaboration2015estimating, baker20161,
pineau2021improving}. In machine learning, however, reproducing
experimental results from a paper alone remains a persistent challenge:
documentation is often incomplete, hyperparameters are under-specified,
and a substantial fraction of accepted papers release no accompanying
code \citep{seo2025papercoder}. This forces researchers to invest
considerable effort reverse-engineering methods from prose, slowing
the pace of scientific dissemination.

Recent Large Language Models (LLMs) have shown strong capabilities in
both understanding scientific text and generating code
\citep{dubey2024llama, openai2024gpt4, reid2024gemini}, prompting a
wave of paper-to-code agents that aim to translate a research paper
directly into an executable repository \citep{seo2025papercoder,
starace2025paperbench, siegel2024corebench}. Systems such as
PaperCoder \citep{seo2025papercoder} decompose this task into
planning, analysis, and coding stages, and emit repositories that are
structurally complete and stylistically convincing. Under
LLM-as-a-judge evaluation, such repositories routinely receive high
faithfulness scores. The implicit conclusion is that the generated
code is a reasonable proxy for the paper's implementation.

We argue that this conclusion conflates three increasingly strict
properties that should be kept distinct: whether a repository
\emph{looks} faithful to the paper, whether it actually \emph{runs},
and whether it \emph{reproduces} the paper's experimental behavior. A
simple analogy is egg fried rice. A finished plate may look right,
but a positive visual judgment does not entail that the cook
understood the recipe, followed the correct procedure, or produced a
dish with the expected taste. A judge that only inspects the plate
cannot detect a missing ingredient, an inverted step, or a heat
profile that differs from the original. The same holds for
paper-to-code systems: a repository may use the wrong loss term, omit
a key normalization, depend on an uninstalled package, or execute
successfully yet fail to match the reported baseline. Existing
benchmarks largely measure surface-level repository quality and do
not separate these three properties, leaving open the question of
whether current paper-to-code agents are reproducing science or
merely producing code that resembles it.

To address this gap, we propose that paper-to-code evaluation should
be \emph{execution-grounded} rather than \emph{appearance-based}, and
we present a prototype framework that realizes this principle .
Given a paper, the framework runs three parallel agents: a
\textbf{coding agent} that emits an initial repository; a
\textbf{rubric and unit-test generation agent} that converts paper
claims into \texttt{pytest} cases targeting intermediate quantities
and final outputs; and a \textbf{runtime-information extraction
agent} that records hardware, package, dataset, and hyperparameter
requirements. A subsequent code-running and amelioration stage uses
rubric checks, test outcomes, and execution traces to revise the
repository. The final artifact is packaged as a TerminalBench-style
\citep{tbench2025} reproducibility task that can be replayed by
external agents. The novelty here is not the multi-agent
decomposition itself, but the shift in what counts as evidence of
reproduction: rather than asking only whether the final result
\emph{looks} right, the framework verifies that the repository
computes the correct intermediate quantities, runs to completion,
and satisfies benchmark-level criteria.

We organize our investigation around three research questions:
\begin{description}[leftmargin=1em,itemsep=0pt,topsep=2pt]
  \item[\textbf{RQ1}] Does LLM-as-a-judge repository quality reflect
    actual runnability and scientific reproducibility?
  \item[\textbf{RQ2}] Can paper-specific rubrics and intermediate-value
    tests better identify algorithmic implementation errors than
    end-to-end output checks?
  \item[\textbf{RQ3}] Can static and runtime feedback meaningfully
    improve generated paper-to-code repositories?
\end{description}

\noindent We make three contributions. \emph{First}, we identify and
characterize a gap between apparent repository quality, executability,
and scientific reproducibility in current paper-to-code agents.
\emph{Second}, we propose a rubric- and execution-guided workflow that
decomposes paper-to-code reproduction into paper understanding, code
generation, rubric and unit-test generation, runtime-information
extraction, static amelioration, and TerminalBench-style task
packaging. \emph{Third}, we evaluate the prototype on 22 papers from
PaperBench \citep{starace2025paperbench} and show that making
generated repositories runnable is substantially easier than
reproducing paper-level experimental claims, underscoring the need
for intermediate-value verification and stronger runtime-guided
refinement.

\section{Related Work}

\paragraph{Paper-to-code generation.} PaperCoder
\citep{seo2025papercoder} is the closest baseline to our work.
It decomposes paper-to-code into planning, analysis, and coding
stages and shows that structured repository generation
improves implementation quality over single-prompt baselines.
However, PaperCoder's main evaluation centers on repository-level
faithfulness as judged by an LLM, with comparatively limited
analysis of executability or experimental reproduction. Our
framework adopts the planning-analysis-coding skeleton but
extends it with paper-specific rubric generation, unit-test
generation, runtime-information extraction, and a code-running
amelioration loop intended to verify the generated repository
against execution-grounded criteria.

\paragraph{Reproduction benchmarks.} A growing body of work
benchmarks paper-to-code systems and code-using agents.
PaperBench \citep{starace2025paperbench} measures whether agents
can reproduce machine-learning papers from scratch using
hierarchical rubrics, and provides the papers we use in our
evaluation. SciReplicate-Bench
\citep{scireplicate2024} focuses on reproducing
algorithmic procedures from method sections.
CORE-Bench \citep{corebench2024} evaluates the reproducibility
of computational research more broadly, while REPRO-Bench
\citep{hu2025reprobench} targets end-to-end reproduction with
explicit execution checks. Compared to these benchmarks, our
focus is not on building a static benchmark but on a
\emph{generative} workflow that turns a paper into a reproducible
task on the fly, while inheriting their key insight that
execution must be part of evaluation.

\paragraph{Agentic terminal benchmarks.} Terminal-Bench
\citep{tbench2025} evaluates agents on realistic terminal tasks
using containerized environments, natural-language instructions,
test suites, oracle solutions, and time limits. Our Harbor-style
packaging adapts this evaluation philosophy to paper-to-code
reproduction: the final artifact is not a static repository but a
self-contained task that bundles paper, code, runtime
specification, and tests.

\paragraph{Test-time refinement.} Iterative self-debugging
\citep{chen2023selfdebug,zhong2024debugbench} and best-of-$N$
sampling approaches \citep{brown2024largelanguagemonkeys} have
shown that test-time feedback can improve LLM code generation.
Our amelioration loop is in this lineage, but uses
\emph{paper-specific} rubrics and intermediate-value tests as the
feedback signal, rather than generic correctness oracles or
arbitrary unit tests.

In this section, we formalize the task of paper-to-code reproduction with paper-grounded evaluation, and then propose our multi-agent framework, which extends repository-level code generation with rubric-driven unit testing, runtime-aware execution, and benchmark packaging.

The goal of our task is to automatically produce a code repository that not only implements the methods and experiments described in a scientific paper, but also satisfies a paper-derived evaluation standard that distinguishes superficially plausible code from code that faithfully reproduces the paper's claims (Pineau et al., 2021; Starace et al., 2025). Formally, given a paper $R$, we define a mapping $\mathcal{M}$ that yields four artifacts:
\[
\mathcal{M}(R) = (C,\, U,\, I,\, B),
\]
where $C = \{c_1, \ldots, c_n\}$ is the generated code repository, $U$ is the paper-specific rubric together with an executable unit-test suite, $I$ is the structured runtime information required to execute the repository (including dataset and environment specifications), and $B$ is a Harbor-style reproducibility task bundle. The most straightforward approach to instantiating $\mathcal{M}$ is to instruct an LLM to generate $C$ alone, as in prior repository-level code generation work \citep{seo2025papercoder}; yet, $C$ alone underspecifies reproducibility, because it conflates the question of whether the code runs with the question of whether the code computes the right quantities. Therefore, we decompose $\mathcal{M}$ into five coordinated stages, with $U$ and $I$ derived directly from $R$ rather than from $C$, so that the paper---not the generated code---remains the source of ground truth. Formally, the overall pipeline is:
\begin{align*}
\text{Code Generation:}\;& (C,\,S) = \mathcal{M}_{\text{code}}(R), \\
\text{Rubric/Tests:}\;& U = \mathcal{M}_{\text{rub}}(R,\,S), \\
\text{Runtime/Dataset Info:}\;& I = \mathcal{M}_{\text{run}}(R,\,S), \\
\text{Amelioration:}\;& C^{\star} = \mathcal{M}_{\text{amel}}(C,\,U,\,I), \\
\text{Packaging:}\;& B = \mathcal{M}_{\text{pkg}}(R,\,C^{\star},\,U,\,I),
\end{align*}
where $S$ denotes a set of structured intermediate artifacts produced during code generation and reused by downstream agents. The overall pipeline is shown in Figure~\ref{fig:workflow}. We then describe how each stage is instantiated below.

\subsection{Code Generation Phase}

The code generation phase is largely the same as PaperCoder \citep{seo2025papercoder}, which we adopt with two modifications. First, we prepend a preprocessing step that converts the input PDF into a cleaned JSON representation $R'$ (\texttt{paper.json}), with citation spans, bibliography entries, headers, and footers removed, so that all downstream agents reason over the same canonical input. Second, in addition to the repository $C$, this phase explicitly exposes a set of structured intermediate artifacts
\[
S = (R',\,P,\,A,\,E),
\]
where $P$ is the planning trajectory, $A$ is the per-file analyses, and $E$ is a paper-grounded extraction containing algorithmic specifications, equations, datasets, baselines, and reported metrics. These artifacts are reused downstream by the rubric agent (for paper-grounded test design), the unit-test agent (for alignment with the planned implementation), and the runtime/dataset agent (for environment and dataset specifications). We note that $S$ is derived from $R'$, not from $C$; conditioning downstream agents on $S$ therefore does not implicitly relax the evaluation standard to whatever the coding agent happened to produce.

Formally, given a paper $R$, we define
\begin{equation*}
\begin{aligned}
\mathcal{M}_{\text{code}}(R) :={}& \mathcal{M}_{\text{debug}} \circ\, \mathcal{M}_{\text{depcode}} \circ\, \mathcal{M}_{\text{ana}} \\
&\circ\, \mathcal{M}_{\text{plan}} \circ\, \mathcal{M}_{\text{pre}}(R) \to (C,\,S).
\end{aligned}
\end{equation*}
where the four core substeps follow PaperCoder closely.

\textbf{Planning.} The planning agent extracts an overall plan, an architecture design, a logic design, and an initial \texttt{config.yaml}, yielding $P = \mathrm{LLM}(\mathcal{T}_{\text{plan}}(R'))$.

\textbf{Per-file analysis.} For each file $f_i$ in the planned repository, the analysis agent writes a natural-language blueprint $a_i$ before any code is emitted, yielding $A = \{a_i\}_{i=1}^{|F|}$.

\textbf{Dependency-aware coding.} Files are generated in dependency order, with $c_i = \mathrm{LLM}(\mathcal{T}_{\text{code}}(R',\,P,\,f_i,\,a_i,\,\{c_1, \ldots, c_{i-1}\}))$, yielding $C = \{c_i\}_{i=1}^{|F|}$.

\textbf{Debugging.} If execution fails during amelioration, the debugging agent proposes minimal search--replace patches rather than rewriting whole files, preserving repository structure and reducing regressions.

We refer the reader to \citet{seo2025papercoder} for full details of the four core substeps. Our contribution in this phase is the additional extraction of $E$ and the explicit downstream interface through $S$, which together enable the paper-grounded rubric, unit-test, and dataset agents described next.

\subsection{Rubric and Unit-Test Generation}

While the code generation phase answers \emph{what} to build, the rubric stage specifies \emph{what successful reproduction means}, and crucially does so without observing the generated repository. Conditioned on the paper $R$ and the structured artifacts $S$ -- but not on $C$ -- the rubric agent produces a structured rubric and an executable \texttt{pytest} suite:
\[
\mathcal{M}_{\text{rub}}(R,\,S) := \mathrm{LLM}(\mathcal{T}_{\text{rub}}(R,\,S)) \to U = (Q,\,V),
\]
where $Q$ is a rubric broken down into code-development, execution, and result-analysis items, and $V$ is the unit-test suite that operationalizes $Q$. Conditioning on $(R, S)$ rather than $C$ is deliberate: if the rubric were conditioned on $C$, the evaluation standard could be implicitly relaxed to fit whatever the coding agent produced, and the paper would no longer serve as ground truth. The structured artifacts in $S$---in particular the equations, algorithmic specifications, and reported metrics in $E$---let the rubric agent ground individual tests in specific equations and quantities of the paper rather than in vague natural-language descriptions. The generated tests fall into two levels.

\textbf{Intermediate-value tests.} These tests check algorithmic internals---loss terms, diffusion coefficients, normalization invariants, tensor shapes, or values derived from specific equations in the paper. They are the diagnostically richest part of the framework: a system can have the right final number for the wrong reason, but it is much harder to silently match a battery of intermediate quantities.

\textbf{Final-comparison tests.} These tests run benchmark-level checks: does the generated method match or outperform a stated baseline on the dataset and metric described in the paper? Final-comparison tests answer the externally visible question, while intermediate-value tests answer the internally faithful question.

We note that this division is the core methodological contribution of our framework: rather than asking ``does the final result look right?'', it asks ``did the implementation compute the important internal quantities correctly, \emph{and} did it also produce the right final result?''.

\subsection{Runtime and Dataset Information Extraction}

Runtime setup is a different problem from code generation, and we treat it as such. Knowing that a paper requires CUDA 12 and a specific Torch version is not the same as knowing how to implement the model, and conflating the two leads to repositories that look correct in isolation but cannot be reproduced. We therefore introduce a dedicated agent that, conditioned on the paper $R$ and the structured artifacts $S$, extracts execution and dataset requirements into a structured artifact $I$:
\[
\mathcal{M}_{\text{run}}(R,\,S) := \mathrm{LLM}(\mathcal{T}_{\text{run}}(R,\,S)) \to I,
\]
where $I$ contains hardware requirements (e.g., GPU memory, CUDA version), package and Docker requirements, datasets and benchmarks (with download endpoints and split specifications), training requirements (e.g., epochs, batch size), hyperparameters and random seeds, and the metrics and reported results to be matched. Reusing the dataset and metric fields already extracted in $E$ avoids re-parsing the paper from scratch, and separating runtime information into its own structured artifact makes it possible to construct, verify, and reuse the environment independently of the code.

\subsection{Code Running and Amelioration}

The amelioration stage is intended to improve the repository using the rubric items in $U$, the runtime information in $I$, and execution feedback. 

\textbf{Static amelioration loop.} The static loop takes the generated repository together with the paper-specific rubric and checks code-development items that can be judged without execution---for instance, that the loss function contains a required penalty term, that the configuration file uses the correct hyperparameter, or that a named module exists at the expected location. When a rubric item fails, the LLM proposes minimal search--replace patches and the loop iterates. Many reproduction errors are visible in source code and can be corrected cheaply at this stage, before any training run is launched.

\textbf{Runtime run-and-fix loop } The full runtime loop should: (1) read $I$; (2) build or configure the environment; (3) download datasets and baselines; (4) run \texttt{reproduce.sh}; (5) execute the generated \texttt{pytest} suite $V$; (6) read \texttt{scores/scores.json}; (7) feed failing tests and logs back to the debugger $\mathcal{M}_{\text{debug}}$; and (8) patch and rerun until tests pass or the iteration cap is reached. The current codebase contains most components in isolation: dataset downloading, debugging via search--replace patches, unit-test execution, and Harbor packaging.



\section{Experimental Setup}

We now describe the experimental setup and the metrics used to distinguish appearance-based from execution-grounded reproduction quality.

\textbf{Papers.} We evaluate the prototype on 22 papers drawn from PaperBench \citep{starace2025paperbench}, spanning a mix of deep-learning, optimization, and benchmark-style contributions. The papers were selected to cover a range of structural complexity and dataset requirements.

\textbf{Backbone.} The backbone model for all agents is GPT-5.2. Calls are routed through the same client and use a shared system-prompt template per stage; only the stage-specific instructions $\mathcal{T}_{\text{stage}}$ and the cleaned input $R'$ differ across calls.

\textbf{Environment.} Each generated repository is packaged into a Harbor-style task with paper JSON, generated code, runtime information, \texttt{pytest} tests, and a weighted reward script. Tasks are run under a 30-minute timeout on a local machine with an RTX 4060.

\textbf{Metrics.} We report three metrics that together separate
superficial faithfulness from execution-grounded reproduction:
(i) \emph{LLM Judge Score}, evaluated by GPT-5.2 against a set of
generated rubrics ;
(ii) \emph{Runnable Rate}, whether the repository can execute without
fatal errors; and
(iii) \emph{Reproduction Score}, a weighted score over code-development,
execution, and result-analysis tests; we do not separately report an
intermediate-value pass rate, since it is subsumed by this score.

\section{Results}

We report preliminary results on a 22-paper subset of PaperBench
together with BE-CBO, used as an in-depth case study. We treat
these numbers as a sanity check on the framework rather than as
a final evaluation; absolute values are likely to shift once the
runtime run-and-fix loop is completed and dataset and baseline
retrieval mature.

The most striking row is runnability. Under our evaluation
protocol, none of the 22 P2C repositories run end-to-end within
the 30-minute timeout, while all 22 P2R repositories do. Two
factors drive this gap: the runtime-information agent fixes most
environment-level mismatches before execution, and the static
amelioration loop catches the structural bugs---missing imports,
mis-spelled module paths, mis-typed configuration keys---that
otherwise prevent a repository from starting. Because no P2C
repository runs, intermediate-value and baseline metrics cannot
be evaluated for P2C, and the corresponding cells are left
empty.

The second row shows that P2R also improves the LLM judge score
on the coding stage from 65\% to 85\%. This is consistent with
the original Paper2Code finding that decomposed generation
produces faithful-looking repositories, and shows that adding
amelioration does not regress on appearance-based quality.

The remaining rows tell a more cautious story. P2R passes
roughly 72\% of intermediate-value tests but matches or beats
stated baselines on only about 15\% of papers. In other words,
even after the framework has produced a runnable repository
whose individual algorithmic components mostly look correct, the
overall system still rarely reproduces the paper's reported
numbers. Two factors contribute. First, part of the
intermediate-value pass rate is achieved at the cost of
correctness: when a generated step fails, the amelioration loop
sometimes patches the failure by simplifying or removing the
offending computation, which keeps the code runnable but lowers
faithfulness on tests it can no longer reach. Second,
baseline-outperformance is sensitive to factors outside the
generated code itself, including dataset retrieval, baseline
availability, and training budget---issues that the static
amelioration loop is not designed to fix and that the runtime
run-and-fix loop is intended to address.

\section{Discussion}

\paragraph{Answering RQ1.} The gap between LLM judge scores and
intermediate-value pass rates supports the claim that
``seems good'' is not enough. A judge that inspects only the
final repository can miss a missing penalty term, an off-by-one
normalization, or an incorrect optimizer setting that does not
prevent the code from running.

\paragraph{Answering RQ2.} Intermediate-value tests are the
strongest diagnostic component of the framework. In several
case studies, an intermediate-value test caught an error that
neither LLM judge inspection nor final metric comparison could
isolate, because the final metric was either similar to baseline
by coincidence or unreachable due to a separate runtime issue.

\paragraph{Answering RQ3.} The static amelioration loop already
provides substantial improvements in code-development quality at
low cost. The full answer to RQ3, however, requires the
completed runtime run-and-fix loop, which we leave to future
work.



\section*{Limitations}

This work has several limitations. First, the runtime run-and-fix
loop is not fully implemented; while the components exist, the
top-level orchestrator that ties them into a single autonomous
process is future work. Second, our experiments cover 22 papers
under a single backbone (GPT-5.2) and a single hardware budget;
broader coverage and multiple-seed runs are needed before any
conclusions about absolute performance can be drawn. Third, our
intermediate-value tests are themselves LLM-generated from the
paper; while we keep them blind to the generated repository,
they may still inherit biases from the backbone, and a careful
audit of test quality is needed. Fourth, dataset and baseline
fetching are not always reliable, and some failures are
infrastructure failures rather than implementation failures.
Fifth, we do not yet run formal ablations isolating the
contribution of each stage.

\section{Future Work}

\textbf{Full runtime orchestrator.} The most immediate next step
is to implement the full runtime run-and-fix orchestrator that
sequences environment build, dataset retrieval,
\texttt{reproduce.sh} execution, test running, log ingestion, and
patch application into a single loop bounded by an iteration
cap.

\textbf{Stronger environment builder.} Many papers require
specific CUDA versions, Torch versions, GPU memory budgets, or
non-Python dependencies. A better environment builder should
translate the runtime-info JSON into a concrete \texttt{Dockerfile}
and installation script, with verifiable post-build checks.

\textbf{Improved dataset and baseline fetching.} Preliminary
results suggest that low reproduction scores are often caused by
dataset retrieval or baseline availability, not by model
implementation errors. Targeted improvements in this area are
likely to raise reproduction scores more than additional
amelioration of model code.

\textbf{Formal ablations.} Important ablations include removing
intermediate-value tests, removing static amelioration, removing
runtime feedback, and comparing against a PaperRepro-only
baseline under matched conditions. These ablations would
quantify the contribution of each component of the framework.

\textbf{Stronger case studies.} BE-CBO illustrates how
intermediate-value tests detect whether key algorithmic
components are correctly implemented. Negative cases are equally
informative, since they show how repositories may be runnable
yet scientifically incorrect, and they argue most strongly for
execution-grounded evaluation.

\section{Conclusion}

PaperRepro asks whether an LLM can generate a faithful-looking
repository from a paper. Terminal-Bench asks whether an agent can
solve realistic terminal tasks under tests. Our project sits
between the two and asks whether a paper-to-code system can
generate a repository that not only looks good, but also runs
and satisfies paper-specific scientific reproduction tests. We
present a rubric- and execution-guided framework that turns a
paper into an initial repository, paper-specific rubrics,
intermediate-value tests, runtime requirements, and a Harbor-style
reproducibility task. On 22 PaperBench papers, the prototype
substantially improves runnability and exposes a large gap between
executable and scientifically reproducible repositories. We view
this as evidence that paper-to-code evaluation should move beyond
LLM-as-a-judge scores toward execution-grounded verification of
algorithmic and experimental fidelity.
